# Fundamental Research Engine 工程介绍

> 一个可复用、主题无关（theme-agnostic）的基本面研究引擎：把一个研究问题固化成一条可重复执行、可审计、可比对的分析流水线。

本文档基于 `origin/main`（最新提交 `156ba8f`，2026-07-02，94 个单元测试全绿）整理，是对整个工程功能特性的完整介绍。

---

## 1. 这是什么

Fundamental Research Engine（FRE）把「行业 / 主题 / 技术趋势」的基本面分析，从一次性的叙事写作，变成一条**确定性的、结构化的、可重复的流水线**。

核心理念：每一个结论都必须回答四个问题——
1. 技术、工作负载、监管、供给或需求**发生了什么变化**？
2. 这个变化制造了**哪个稀缺资源 / 瓶颈**？
3. 价值链的**哪一环能捕获**这个瓶颈的价值？
4. **什么证据**能证实或证伪这个论点？

它刻意**不做**短期价格目标和详细估值，只聚焦结构性的产业逻辑、瓶颈归属和价值捕获。

### 关键定位
- **主题无关**：AI 只是第一个领域包和第一批样例主题。电池、金属、医疗、消费、政策、宏观、地缘同样可以用「领域包 + 方法论包」表达。
- **零外部依赖**：纯 Python 标准库实现（`pyproject.toml` 中 `dependencies = []`）。连模型调用和网页抓取都用 `urllib` 手写，CI 全程离线可跑。
- **结构化 JSON 是唯一事实来源**：模型只产出固定 schema 的分阶段 JSON，绝不直接写备忘录散文；产出后仍要通过校验。
- **可审计优先于自动化**：先把确定性内核和证据审计做稳，再谈自动采集。

## 2. 通用分析流水线

无论什么主题，都走同一条链条：

```text
主题定义 (theme definition)
  -> 驱动分类 (driver classification)
  -> 传导机制链 (mechanism chain)
  -> 成熟度评估 (maturity assessment)
  -> 瓶颈诊断 (bottleneck diagnosis)
  -> 价值链映射 (value-chain mapping)
  -> 利润池分析 (profit-pool analysis)
  -> 公司卡位 (company positioning)
  -> 证据与反论点追踪 (evidence & counter-thesis)
  -> 情景分析 (scenario analysis)
  -> 备忘录生成 (memo generation)
  -> 跑次间比对 (run-to-run diff)
```

### 支持的主题类型（方法论包）
每种类型对应 `knowledge/methodologies/` 下的一个包，规定该类主题该强调什么：

| 主题类型 | 适用场景 | 强调重点 |
|---|---|---|
| `technology_adoption` | AI、机器人、固态电池、CPO、新材料、商业航天 | TRL、Hype Cycle、S 曲线、学习曲线、客户认证、生态就绪度 |
| `supply_demand_cycle` | 资源、内存周期、航运、电力设备、变压器 | 库存、产能扩张周期、开工率、价格弹性、替代、需求破坏 |
| `policy_driven` | 低空经济、国防、国产化、碳政策、医改、补贴 | 政策工具、预算流、执行主体、采购路径、执行力度 |
| `consumer_adoption` | 消费品牌、渠道变迁、人口结构、行为变化 | 渗透率、频次、定价/结构、渠道经济、留存、品牌持久性 |
| `healthcare_clinical` | GLP-1、ADC、细胞治疗、医疗器械、临床平台 | 疗效、安全性、临床阶段、标准疗法、可及人群、监管路径、支付 |
| `macro_cycle` | 黄金、银行、地产、券商、出口商、利率敏感资产 | 利率、信用、汇率、通胀、财政脉冲、资产负债表、风险偏好 |
| `geopolitics_security` | 国防、能源安全、供应链本地化、粮食安全、战略储备 | 情景、预算、咽喉点、出口管制、本地化、储备 |

### 采用的分析框架
- **Mosaic research**：多个合法可归属来源合成论点，每条证据都带来源日期、类型、声明、可靠度。
- **Gartner Hype Cycle**：区分「叙事温度」和「部署成熟度」（`trigger → inflated_expectations → disillusionment → enlightenment → productivity`）。
- **NASA TRL**：1-9 技术就绪度打分，区分「技术可行」和「商业部署」。
- **Porter 五力**：判断价值捕获和议价能力。
- **Thematic investing**：区分结构性趋势和短期风口，拒绝没有产品级敞口的宽泛标签。
- **Wright's Law（学习曲线）**：定性推理成本下降和规模效应，关注瓶颈何时被商品化。

方法论细节见 `docs/methodology.md`。

## 3. 数据模型（结构化契约）

所有研究对象都是 `src/fundamental_research_engine/models.py` 里的冻结 dataclass。一个主题（`Theme`）由以下部分组成：

- **主题元信息**：`id`、`title`、`as_of`（日期）、`theme_type`、`domain`、`core_question`、`thesis`、`mechanism`、`hype_stage`、`technology_readiness_level`、`drivers`
- **Bottleneck（瓶颈）**：带一张 8 维打分卡 `scorecard`，链接支撑证据 `evidence_ids`
- **Segment（价值链环节）**：层级、角色、受益类别、代表公司
- **ProfitPool（利润池）**：捕获质量、受益方、逻辑
- **CompanyPosition（公司卡位）**：产品、栈内位置、卡位标签、敞口质量、护城河、风险、证据
- **Scenario（情景）**：描述、推论、触发条件
- **Evidence（证据）**：来源类型、日期、URL、可靠度、具体声明列表
- **反论点** `counter_theses` 与 **追踪信号** `tracking_signals`

除 `Theme`/`Evidence` 外的对象都支持**稳定 `id`**（缺省回落到 `name`），这是让跨跑次 diff 能识别「改名 vs 增删」的关键。

### 瓶颈打分机制（`scoring.py` + `knowledge/scoring_rules.json`）
8 个维度分为正向和负向两组，加权平均：
- **正向**（分值越高瓶颈越强）：需求增速、产能扩张难度、技术替代难度、良率/材料/设备约束、客户认证锁定、供应商定价权
- **负向惩罚**：快速放量风险、架构绕过风险

最终分 = `正向加权分 − 负向加权分 × 0.35`，钳制在 0-5，再映射到 `critical / strong / watch / weak` 评级。

> ⚠️ 重要原则：打分是**过程健康度信号**，不是「论点为真」的分数。证据覆盖率同理——衡量研究流程是否扎实，而非论断正确。

## 4. CLI 命令全景

入口是 `fre`（`pyproject.toml` 注册），或 `PYTHONPATH=src python3 -m fundamental_research_engine <cmd>`。

| 命令 | 作用 |
|---|---|
| `run <theme>` | 跑完整流水线，输出 `analysis.json` + `memo.md` |
| `validate <theme>` | 只做 schema 校验，不跑流水线 |
| `audit <theme> [--out]` | 只构建证据审计报告 |
| `evidence-sync <theme> [--fetch-sources]` | 把证据同步进本地证据库；加 `--fetch-sources` 真正抓 URL |
| `split <theme.json> <dir>` | 把单文件主题拆成 6 个分阶段文件 |
| `merge <dir> <out.json>` | 把分阶段目录合回单文件 |
| `diff <theme_id>` | 比对某主题最近两次跑次（或用 `--from/--to` 指定） |
| `fill <dir>` | 用模型适配器起草下一个缺失阶段 |
| `draft <dir> [--auto]` | 多阶段编排；默认每阶段停下等审阅，`--auto` 无人值守走完 |
| `critique <dir> --stage <s>` | 对已起草的某阶段做对抗式评审 |

### 输出产物
- 跑次结果写在 `runs/<as_of>-<theme_id>/`：
  - `analysis.json`：结构化流水线输出（含 `evidence_audit` 段）
  - `memo.md`：人类可读的研究备忘录
- diff 结果写在 `runs/diffs/<theme_id>-<from>-to-<to>/`（`diff.json` + `diff.md`）
- `runs/` 和 `data/` 下生成物默认被 git 忽略（只留 `.gitkeep`）。

## 5. 三种作者工作流

### 5.1 单文件手写
最直接：一个主题就是一个 JSON 文件（如 `configs/themes/hbm4.json`），直接 `fre run`。

### 5.2 分阶段编写
一个主题可以拆成 6 个分阶段 JSON 文件（对应流水线的六大步），放在一个目录里：
```
theme_definition.json / mechanism_analysis.json / bottleneck_diagnosis.json
value_chain_map.json / company_positioning.json / scenario_analysis.json
```
`run`/`validate`/`diff` 都同时接受单文件和目录两种形态，且输出**逐字节一致**。`configs/themes_staged/hbm4/` 就是从单文件拆出的范例。

拆分的意义：让每个阶段成为**小而可独立评分的工作单元**，这是 LLM 辅助编写的前提。

### 5.3 LLM 辅助起草
`prompts/<stage>.md` 是每个阶段的提示词模板，内嵌该阶段所需 JSON schema、本体枚举、对应主题类型的方法论包，以及所有已完成的上游阶段作为上下文。

- **`fre fill`**：起草第一个缺失阶段。
  - 默认「手动/离线」模式：把提示词写到 `<stage>.prompt.md`，你拿去任意模型 UI 跑，把 JSON 回复存成 `<stage>.json`。
  - `--model claude/openai --model-name <id>`：直接调 API（key 从环境变量读）。
  - 强健性：能从纯 JSON、markdown 代码块、夹带散文中提取 JSON 对象；shape 不对会带着校验错误重试（`--max-attempts`，默认 2）；每个被接受的阶段生成 `<stage>.meta.json` 审计记录（模型、尝试次数、提示/回复哈希、时间戳）。
- **`fre draft`**：在 `fill` 之上编排多阶段。
  - 默认是**带检查点的工作流**：起草一个阶段后停下，提示你审阅后再重跑——研究内容里，每阶段的人工检查点能在错误蔓延到下游前拦住它。
  - `--auto`：无人值守连走所有剩余阶段，最后自动 `validate` + `run`（拒绝 `--model manual`，因为手动模式没法无人值守）。
- **`fre critique`**：对某个已起草阶段做**第二次对抗式模型评审**。评审模型被要求默认持怀疑态度，专挑无支撑的打分、稻草人反论点、含糊的追踪信号、与上游阶段的不一致。产出 `<stage>.critique.json`（带严重度的 `concerns`、`overall_assessment`、`accept/revise` 建议）。这是**独立检查**，不会自动阻断或改写 `fill`/`draft` 的产出——由人决定是否采纳。

> 贯穿始终的契约：模型永远只产出某一阶段固定 shape 的 JSON，之后仍要过 `validate`/`run`。结构化 JSON 始终是事实来源。

## 6. 证据子系统

目前是一个**审计层**，而非自动采集器。

### 证据审计（`build_evidence_audit`）
`fre run` 会把 `evidence_audit` 嵌入 `analysis.json`，`memo.md` 也有对应章节。审计产出：
- 按来源类型和可靠度的**来源清单**
- 从证据项到其具体声明的**声明链接**（如 `E1.C1`）
- 瓶颈和公司层级的**证据覆盖率**
- **缺失证据引用**
- **覆盖健康分**和状态（`missing / partial / thin / adequate / strong`）

### 本地证据库（`fre evidence-sync`）
把主题证据落盘成确定性的本地记录：
```
data/raw_sources/<theme_id>/<evidence_id>.json  # 来源快照
data/normalized/<theme_id>/evidence.json        # 归一化证据（含稳定声明 id + 关联 owner）
data/evidence/<theme_id>/claims.json            # 声明记录（如 E1.C1）
data/evidence/<theme_id>/coverage.json          # owner 级覆盖率
data/evidence/<theme_id>/audit.json             # 完整审计报告
data/evidence/<theme_id>/manifest.json          # 文件清单 + 抓取结果
```

### 真实抓取（`--fetch-sources`）
默认关闭。开启后 `default_fetch` 会真正抓取每条证据的 URL：
- 仅限 http/https；先查 `robots.txt`（`urllib.robotparser`，不可达时按爬虫惯例默认放行）
- 限制超时和响应大小；无外部依赖地把 HTML 剥成纯文本
- **任何失败**（robots 禁止、网络错误、超时）都对该条**回落到配置快照**，不会让整次 sync 失败；`manifest.json` 逐条记录抓取结果

> 尚未做：把抓取到的原始文本自动抽取成结构化声明的 LLM 阶段——目前抓取只存文本，仍需人工或未来阶段填进 `evidence[].claims`。

## 7. 跑次间比对（Diff）

`fre diff` 是「可重复研究」的闭环：主题更新 `as_of` 和证据后重跑，就能看到到底变了什么。

`diff_analysis(old, new)` 比对两份 `analysis.json`：主题标量字段、驱动、瓶颈分数、价值链环节、利润池、公司、情景、证据、反论点、追踪信号。每个 section 报告 `added/removed/changed`，带字段级 before/after。

对象优先按**稳定 `id`** 匹配，回落到显示名，所以「改名」显示为字段变化而非增删噪音。渲染的 markdown 会同时给出可读标签和 id，如 `SK hynix [co-sk-hynix]`。

```text
runs/                   跑次产物（git 忽略）
reports/                精选报告输出（git 忽略）
docs/                   方法论与设计笔记
tests/                  单元测试（含 golden memo 快照）
```

## 11. 设计取舍与现状

**已完成的里程碑**（详见 `PROJECT_CONTEXT.md`）：确定性 CLI 内核 → schema 校验 → 跑次 diff → 分阶段输入 → LLM 提示词模板 → 多模型适配器 → 稳定 id → 证据审计 → 本地证据库 → CI/golden 快照 → `draft`/`critique` 编排 → 真实证据抓取。

**主要风险与纪律**：不要对定性研究过度打分。证据覆盖率和瓶颈分都是**流程健康信号**，不是论点真值。

**尚未做（有意推迟，非遗漏）**：
- GitHub 仓库的分支保护 / 必需检查（属共享基础设施设置，需仓库 owner 确认）。
- 把抓取的原始文本自动抽成结构化声明的 LLM 阶段。

**协作约定**：本仓库已有多个独立 agent 会话（Claude 与 Codex）直接向 `main` 提交。当可能并发时，建议每会话用短生命周期特性分支，合并前拉取 `origin/main` 并重跑测试。新会话开工前请先读 `PROJECT_CONTEXT.md`。
